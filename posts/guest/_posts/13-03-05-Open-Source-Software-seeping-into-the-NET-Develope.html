Question: Tell us about your company and your relationship with the .NET community.

Patrick Hynds: I have been part of the .NET community since I first learned about the technology before it was even released. We provide development and security consulting in addition to building commercial software products. While we do work with a wide range of technologies, I personally consider .NET my primary platform and when I speak at user groups and Code Camps it is always either on security or .NET.

Q: Why have .NET developers been slow to adopt open source? Many of them operate in the mixed Windows/Linux world, what was the issue?

PH: The open source movement is not incompatible with the Microsoft development world, but the relationship between the two had been hostile in that both view the world differently at the macro level.  For example, commercial software developers represent a big constituency among the .NET Framework community and based on my experience these shops are very cautious about incorporating open source libraries because the licensing (e.g. the GNU General Public License) can influence how their entire solution is licensed and sold. Uncertainty and a lack of understanding of the legal implications have fed this hesitancy.

This uncertainty sets the stage for avoidance of open source components by a large part of the .NET developer population. In fact I had seen developers fired because they included a component into a project without getting clearance to ensure the license of that component did not impinge on the licensing of the project as a whole.

Q: But the GPL license isn’t the only option for FOSS projects, is there more comfort with less restrictive licenses, such as BSD and Apache?

PH:  Part of the issue is educating the legal community that has been very focused on protecting their client’s sales model and IP. With big open source projects becoming mainstream on Windows, more and more developers have realized that there are open source licenses that can work with their business model. In many cases this has just been a learning curve that business people and their attorneys needed to work through. 

Q: There was a lot of FUD about security early on, is that still a concern?

PH: Many windows developers had been told that because anyone can contribute that the code in open source projects can contain back doors while the open source crowd holds the position that more eyes on the code brings greater transparency and therefore better security. The truth is that in most cases open source components are safe provided the keeper of the final code is vigilant. The safest way to proceed is to pay attention to who controls the code in the project. In reality anyone can submit code to an open source project, but that does not mean that that code is accepted and incorporated into the final libraries.  

Q: What areas are you seeing more interest in by .NET developers?

PH: A light at the end of this tunnel appears to be NuGet. NuGet, an open source package management system was built by Microsoft (assigned to the Outercurve Foundation), and ultimately integrated into Visual Studio. Since its introduction I have seen a big jump in the willingness of Microsoft development shops to try open source libraries and include them in their projects. Visual Studio development is a very productive, comfortable, and comprehensive environment. It has historically required developers to do little to nothing to knock out a site or system and spares the developer from tracking down the correct component versions of their environment which is a regular experience for java developers. NuGet helps developers have their cake and eat it too in regards to getting packages and libraries for just about everything without the strain typically associated with composing a solution from many packages and their dependencies. 

I think this one factor will see a lot more open source finding its way into Microsoft-based development, and while some may still steer clear of the GPL, there are dozens of FOSS licenses that are compatible with Windows developers and their customers.
