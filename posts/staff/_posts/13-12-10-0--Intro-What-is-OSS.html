This post is the first in a series that will explore why businesses should become active participants in open source communities. We’ll explore what open source is, the economics of OSS (open source software), who uses it, how to make projects successful, and how to participate in external projects. It will also taken a deep dive into more complex topics such as the commercialization of FOSS, community vs customers, managing inbound IP, licenses, software hygiene, and when NOT to publish a project.

Software is surprisingly dynamic. All software evolves. Bugs are found and fixed. Enhancements are added. New requirements are discovered in using the software.  New uses are found for it, and it is shaped to those new uses. It doesn’t matter whether the software under discussion is an internal IT project, a commercial proprietary product or a collaborative public software project that is liberally licensed.

Creating good software is hard work. People have shared software and collaborated on its creation for as long as it has existed, all the way back through conferences run by the computer vendors (e.g. DEC and DECUS, IBM and SHARE). The rise of the Internet and the World Wide Web has removed the friction of calendars and location from these collaborative efforts, accelerating the sharing and collaboration that has always existed.  

During the 1970s and 1980s the University of California, Berkeley and MIT were central in liberally licensing software that they developed collaboratively with the computer vendors, and the licenses they defined are still in use today. The X Windows System and the Berkeley UNIX system variations were all developed and licensed this way.

In the mid-1980s, Richard Stallman developed a definition for software freedom as a way to license and publish software as a public good. The definition is focused on a set of freedoms a software user must have.  Stallman pioneered a software copyright license called the General Public License (GPL). The GPL continues to evolve under the auspices of the Free Software Foundation (FSF), a non-profit organization he created to promote software freedom. The FSF maintains a list of other licenses that support its definition of free software, and an excellent discussion on how the GPL interacts with other liberal licenses.

Meanwhile, in the late 1990s, a group of commercially-focused individuals assembled to debate and define a description and terminology for these disparate efforts to license collaboratively-developed software, and the Open Source Definition (OSD) was born. It specifies a set of attributes that a software license must support to be considered an open source software license. This led to the formation of the Open Source Initiative (OSI), a non-profit organization that maintains the OSD and decides whether liberal software licenses meet the definition. The OSD was designed to encompass the free software definition, but goes beyond it. In other words, all free software can be considered open source software (OSS), but not all open source licensed software supports the FSF definition of software freedom. 

So free and open source software (FOSS) is a licensing mechanism on software.  It encourages the use and re-use of software. Much of the history of software development and software engineering is focused on reliably creating good software, and most would agree that software re-use in all its guises is a good thing. When you use OSS licensing with well-organized collaborative software communities, you create the right conditions to make this use, re-use, and dynamic evolution of software successful. 

This guide is organized into three sections. The first begins by exploring the economics of why shared collaboration works, goes on to explain how individuals buy, use, and make FOSS, and concludes with a discussion of how companies can make FOSS. The second section  delves into greater detail about the development of successful open source projects, communities, and products.  The third and final section presents a discussion about how FOSS foundations enable the growth of FOSS projects.
