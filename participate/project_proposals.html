The Outercurve Foundation Project Acceptance Process

Assigning a project to the Outercurve Foundation provides a number of key benefits. The Foundation acts as a neutral not-for-profit third party encouraging contributions. The IP management practices provided through the Foundation can encourage others to adopt the project. The Foundation also provides marketing and communications services for the project to help get the word out.

The Outercurve Foundation provides a general framework for projects based on the following principles:

  • Open Source Licensed: The Foundation accepts community-based collaborative projects licensed using any Open Source Initiative approved license.
  • Community Developed: The Foundation software development and project management practices exist to support good software IP management and to foster a growing community without being tied to particular tools or forges.
  • Technology Agnostic: The Outercurve Foundation is not tied to a particular technology base and is an efficient and effective way for a software collaborative community or organization to capture the legal benefits and experience of a foundation without starting one from scratch.

Creating a Project

The Foundation project creation process starts with a proposal. It's a simple three step process.

  1. Download the Outercurve Foundation Project Proposal (DOCX).
  2. Fill out the first page of the proposal per the following instructions.
  3. Send it to projects@outercurve.org.

The Foundation will assign a proposal mentor to the project that will contact the proposer and work with them to complete the Extended Project Acceptance Criteria on the proposal. The completed proposal form will be submitted to the appropriate Outercurve Foundation Gallery Manager for approval and then to the Board of Directors for review and acceptance of the project into the Foundation.

Filling out the beginning of the Project Proposal Form

The following information is required on the Outercurve Foundation Project Proposal Form to start the process:

  • Short description: The abstract. What's the one or two sentence paragraph that sums up the project.
  • Long description: The longer description. This section provides more in depth information about the project, its history, context, and background.
  • License: What OSI approved license(s) will the project use for outbound license.
  • URL: The project URL(s)
  • Submitter: Submitter name, email, and affiliation.
  • Project contact: If different from submitter (name, email, affiliation).
  • Project Mentor: Assigned by the Foundation.
  • Committers: List the name, email, and affiliation of at least three committers on the project.
  • Proposed Gallery: Where does the project naturally fit within the gallery structure of the Foundation.

A proposal mentor will be assigned by the Foundation. It may be the gallery manager for a project which is being encouraged by a gallery manager or that is requested to fit into a particular gallery. It may also be a board member, staff or a Foundation advisor.

Filling out the Extended Project Acceptance Criteria

The mentor will contact the project submitter and together they will collect further information about the project for presentation to the Foundation Board. This information includes:

  • Initial Goals: A brief statement of the project's initial goals under the Foundation.
  • Background: Project history and background. When was it started? By whom? Is there a product or research basis to the work.
  • Project Alignment: Why this gallery? Does the project align particularly well with other Foundation projects? Other external projects? Overlap with others? Take a novel approach compared to similar projects?
  • Open Source Experience: What open source community experience do the initial committers and project leader have?
  • Naming Check: Project name check against search engines and the USPTO.
  • Project Committer Management: While the Foundation does not dictate how a project runs its community, or how new committers are promoted, the Foundation does want to understand the basic ideas the initial project lead and committers are going to use to manage their project.
  • Community Development: What does the project hope to accomplish with it's community? How do they plan on developing/growing/encouraging the community? Are there specific services they will need from the Foundation (e.g. IRC channel)?
  • Development Plan: Forge and URL? Code repository and URL? Bug tracking tools and URL? Will the project require code-signing services?
  • Initial Source: From where does the initial code originate? Code history? Are there multiple code bases coming together?
  • IP Plan: Does the project plan to use assignment or license agreements from contributors? Discuss. If the project is being bootstrapped from multiple outside entities, what's the plan to merge them together under the project's banner?
  • Legal Issues: Are there known legal concerns or issues? For example, does the project implement a standard that may make IP demands?
  • External Dependencies: Does the project depend upon other open source software projects outside of its own code base? Are the licenses compatible?
  • Cryptographic Issues: Does the code involve any crypto issues directly or indirectly that will require the Foundation to take steps on behalf of the project?
  • Known Risks: Are there issues that need consideration? It doesn't mean that a project with "known risks" isn't acceptable so much as it demonstrates the people proposing the project understand how the project impacts the larger world. For example a reasonable "risk" is that the participants have no real open source experience. Discussion here means a stronger mentor may need to be involved by the Foundation.

This information gives the Foundation Board a strong feeling for the initial project community and their experience, how the project meets the goals of the Foundation, and how strong a mentor the project may need to help them succeed. It also assures the Board, Gallery Managers, and Sponsors that the IP management process is properly in place and the project is organized to support it.

The information will be circulated to the Foundation Board that will vote to accept the project. If the vote is successful, the Foundation staff will identify and assign a full time mentor to the project.

Once a Project is Accepted by the Foundation

Once the Outercurve Foundation accepts a project, the project leader, project mentor and gallery manager work together with Foundation staff to move the project into the Foundation. This happens through several concurrent tasks, some of which may begin during the acceptance process:

  • Initial IP Assignment: The project manager starts the process to get the initial assignment agreements for the project software IP signed by the appropriate IP holders. Foundation staff and legal counsel help with the process. Please see Assignment Agreements and Contribution License Agreements for a discussion about Foundation assignment agreements.
  • Contribution Agreements: The Foundation office works with the project manager to get contribution agreements from initial committers in place. Please see Assignment Agreements and Contribution License Agreements for a discussion about Foundation contribution agreements.
  • The Communications Plan: The Foundation office works with the project manager to put a communications plan in place to announce the acceptance of the project and encourage community engagement.
  • Services Plan: The Foundation staff work with the project manager to organize any Foundation services the project will want to take advantage of (e.g. code signing).
